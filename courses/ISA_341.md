### Relational database design

**Definition:** Relational database design is the process of organizing data into structured tables (relations) where each row represents a unique record and each column represents an attribute, interconnected through defined relationships. Rooted in E.F. Codd's relational model (1970), it ensures logical and physical data independence so that schema changes do not break the application layer. Good relational design minimizes redundancy, enforces data integrity through constraints, and enables efficient querying.

**Key Concepts:**
- **Primary Keys:** Unique identifiers for each row in a table, ensuring data integrity and efficient retrieval. Candidate keys are all possible unique identifiers; one is chosen as the primary key.
- **Foreign Keys:** References to primary keys in other tables that establish relationships and enforce referential integrity -- the database will reject orphaned references automatically.
- **Relationship Types:** One-to-one (1:1), one-to-many (1:N, the most common), and many-to-many (M:N, resolved via junction/associative tables that decompose the relationship into two one-to-many links).
- **Data Integrity Constraints:** Domain constraints limit valid values for a column, entity integrity ensures no primary key is null, and referential integrity ensures foreign keys point to existing records.
- **Relational Operations:** The core operations -- SELECT (filter rows), PROJECT (choose columns), and JOIN (combine tables) -- form the theoretical basis for SQL queries.
- **Design Goals:** Eliminate data redundancy, prevent insertion/update/deletion anomalies, and ensure the database accurately models real-world entities and their associations.

**Practical Applications:** Relational database design is used in virtually every enterprise system -- from e-commerce platforms managing product catalogs and customer orders, to banking systems tracking accounts and transactions, to healthcare systems linking patients with medical records.

### Normalization

**Definition:** Normalization is a systematic process of organizing a relational database schema into progressively stricter "normal forms" to reduce data redundancy and prevent anomalies. Originally introduced by E.F. Codd in 1970, normalization decomposes large, flat tables into smaller, well-structured tables linked by relationships. A properly normalized database ensures data consistency, simplifies maintenance, and prevents insertion, update, and deletion anomalies.

**Key Concepts:**
- **First Normal Form (1NF):** Each column contains only atomic (indivisible) values, each row is uniquely identifiable, and there are no repeating groups or arrays within a single cell.
- **Second Normal Form (2NF):** Satisfies 1NF and eliminates partial dependencies -- every non-key attribute must depend on the entire composite primary key, not just part of it. Tables with a single-column primary key automatically satisfy 2NF if they are in 1NF.
- **Third Normal Form (3NF):** Satisfies 2NF and removes transitive dependencies -- non-key attributes must depend only on the primary key and not on other non-key attributes. For example, if a table stores CustomerID, CustomerCity, and CustomerZip, the city depends on the zip code (a non-key attribute), violating 3NF.
- **Boyce-Codd Normal Form (BCNF):** A stricter version of 3NF requiring that every non-trivial functional dependency has a superkey on its left-hand side. Every BCNF relation is in 3NF, but not every 3NF relation is in BCNF.
- **Anomaly Prevention:** Without normalization, inserting new data may require redundant entries (insertion anomaly), updating one copy but not others causes inconsistency (update anomaly), and deleting a record may unintentionally remove unrelated data (deletion anomaly).
- **Denormalization Trade-off:** In practice, databases are sometimes intentionally denormalized for read-heavy workloads (e.g., data warehouses, reporting tables) to improve query performance at the cost of some redundancy.

**Practical Applications:** Normalization is applied during the schema design phase of any transactional database -- for example, splitting a flat "Orders" spreadsheet into separate Customers, Products, and OrderDetails tables so that a customer's address update only needs to happen in one place.

### SQL

**Definition:** Structured Query Language (SQL) is the standard programming language for managing and manipulating relational databases. Developed in the 1970s at IBM by Chamberlin and Boyce and later standardized by ANSI/ISO, SQL provides a declarative syntax for defining database structures, querying data, and controlling access. Nearly all relational database management systems (Oracle, MySQL, SQL Server, PostgreSQL) use SQL as their primary interface.

**Key Concepts:**
- **Data Definition Language (DDL):** Commands that define or modify database structure -- CREATE (build tables, views, indexes), ALTER (modify existing structures), DROP (remove objects), and TRUNCATE (clear all rows from a table while preserving structure).
- **Data Manipulation Language (DML):** Commands for working with data -- SELECT (retrieve records with filtering, sorting, and grouping), INSERT (add new rows), UPDATE (modify existing rows), and DELETE (remove rows).
- **Data Control Language (DCL):** Commands that manage permissions and access -- GRANT (give privileges) and REVOKE (remove privileges) control who can perform which operations on which objects.
- **Transaction Control Language (TCL):** Commands like COMMIT, ROLLBACK, and SAVEPOINT manage groups of DML statements as atomic units, ensuring data consistency even during failures.
- **Joins and Subqueries:** SQL supports INNER JOIN, LEFT/RIGHT/FULL OUTER JOIN, and CROSS JOIN to combine data from multiple tables, as well as nested subqueries and Common Table Expressions (CTEs) for complex filtering and aggregation.
- **Aggregate Functions and Grouping:** Functions like COUNT, SUM, AVG, MIN, and MAX combined with GROUP BY and HAVING clauses enable powerful data summarization and reporting.

**Practical Applications:** SQL is used daily by data analysts writing ad-hoc reports, backend developers building application queries, database administrators managing schema migrations, and business intelligence teams creating dashboards from enterprise data warehouses.

### ER diagrams

**Definition:** An Entity-Relationship (ER) diagram is a visual blueprint that illustrates how entities (such as people, objects, or concepts) relate to each other within a database system. Introduced by Peter Chen in 1976, ER diagrams use a standardized set of symbols -- rectangles for entities, diamonds for relationships, and ovals for attributes -- to map out the logical structure of data before physical implementation. They serve as the primary communication tool between business stakeholders and database designers during the requirements-gathering phase.

**Key Concepts:**
- **Entities and Entity Sets:** Entities are real-world objects or concepts (e.g., Student, Course) represented as rectangles. Each entity set groups all instances sharing the same attributes. Weak entities (double-bordered rectangles) cannot be uniquely identified without a related "owner" entity.
- **Attributes:** Properties that describe an entity (e.g., StudentID, Name) shown as ovals. Attributes can be simple, composite (divisible into sub-parts), derived (calculated), or multivalued; key attributes (primary keys) are underlined.
- **Relationships:** Associations between entities depicted as diamonds (e.g., "Enrolls" connects Student and Course). Relationships can have their own descriptive attributes, such as an enrollment date or grade.
- **Cardinality Ratios:** Define how many instances of one entity relate to another -- one-to-one (1:1), one-to-many (1:N), or many-to-many (M:N). These ratios directly influence how foreign keys and junction tables are implemented in the physical schema.
- **Participation Constraints:** Total participation (double line) means every instance of an entity must participate in the relationship; partial participation (single line) means participation is optional. These constraints affect NOT NULL requirements in the physical database.
- **Diagram Levels:** Conceptual ERDs capture high-level business entities and relationships; logical ERDs add attributes and keys; physical ERDs specify data types, indexes, and platform-specific details.

**Practical Applications:** ER diagrams are used at the start of every database project to translate business requirements into a visual schema -- for instance, mapping out the relationships between Departments, Employees, and Projects before writing a single line of SQL. Tools like ERDPlus, Lucidchart, and draw.io facilitate collaborative creation.

### Database application development

**Definition:** Database application development is the practice of building software systems that interact with a database to store, retrieve, and manipulate persistent data. At its core, it revolves around implementing CRUD operations (Create, Read, Update, Delete) through an application layer that communicates with the database via SQL or an Object-Relational Mapping (ORM) framework. Effective database application development balances performance, security, and maintainability across multiple architectural tiers.

**Key Concepts:**
- **CRUD Operations:** The four fundamental operations map directly to SQL -- INSERT (Create), SELECT (Read), UPDATE (Update), and DELETE (Delete). Every data-driven application is built on these operations, and HTTP methods in REST APIs correspond directly: POST, GET, PUT/PATCH, DELETE.
- **Stored Procedures and Functions:** Precompiled SQL code stored on the database server that encapsulates business logic, improves performance by reducing network round-trips, and enhances security by limiting direct table access to application users.
- **Transaction Management (ACID):** Transactions group multiple operations into atomic units governed by ACID properties -- Atomicity (all or nothing), Consistency (valid state transitions), Isolation (concurrent transactions don't interfere), and Durability (committed changes persist). Commands like COMMIT, ROLLBACK, and SAVEPOINT control transaction boundaries.
- **Connection Management and Pooling:** Applications use connection pools to reuse database connections efficiently, reducing the overhead of establishing new connections for each request and improving scalability under high concurrency.
- **Security Practices:** Parameterized queries and prepared statements prevent SQL injection attacks. Role-based access control (RBAC) and the principle of least privilege limit what each application user or service account can do.
- **ORM Frameworks:** Tools like Hibernate (Java), Entity Framework (.NET), and SQLAlchemy (Python) map database tables to application objects, allowing developers to interact with data using programming language constructs rather than raw SQL.

**Practical Applications:** Database application development powers virtually all modern software -- from web applications with user registration and content management, to point-of-sale systems processing retail transactions, to enterprise resource planning (ERP) systems coordinating inventory, finance, and human resources.

### Data modeling

**Definition:** Data modeling is the process of creating a visual representation of a system's data elements and the relationships between them, serving as the blueprint for database design and implementation. The process follows three progressive levels of abstraction -- conceptual, logical, and physical -- each adding more technical detail as the design moves from business requirements toward an implementable database schema. Data modeling ensures that all stakeholders share a common understanding of how information is structured and flows through the system.

**Key Concepts:**
- **Conceptual Data Model:** The highest-level, technology-independent view that captures business entities and their relationships without specifying data types or constraints. Typically uses ER diagrams to communicate with non-technical stakeholders during requirements gathering.
- **Logical Data Model:** Translates the conceptual model into a detailed structure with defined tables, columns, primary/foreign keys, and relationship constraints, but remains independent of any specific database platform. Used by analysts and architects to validate business rules.
- **Physical Data Model:** The implementation-ready design specifying exact database-specific details -- storage engines, indexes, partitioning strategies, data types, and tablespace configurations optimized for performance on a chosen DBMS (e.g., Oracle, SQL Server, PostgreSQL).
- **Forward and Reverse Engineering:** Forward engineering generates a physical database from a data model; reverse engineering extracts a data model from an existing database, which is essential for documenting and understanding legacy systems.
- **Common Notations:** Chen notation (ovals, rectangles, diamonds), Crow's Foot / IE notation (lines with fork/circle/bar endings indicating cardinality), and UML class diagrams are widely used conventions for representing entities, attributes, and relationships.
- **Data Governance and Quality:** Data modeling supports governance by documenting data lineage, ownership, and business rules, ensuring consistency and quality across the organization and facilitating regulatory compliance.

**Practical Applications:** Data modeling is essential in enterprise projects such as designing a data warehouse that consolidates sales, marketing, and finance data, or migrating a legacy system to a modern cloud database where a clear logical model ensures nothing is lost in translation.
