# ISA 441: Systems Analysis and Information Technology Consulting

**Credits:** 3

### Systems analysis

**Definition:** Systems analysis is the systematic investigation of a business problem or opportunity, involving rigorous understanding of the existing system to identify requirements and characteristics for a new or improved information system. It bridges business needs and technology solutions.

- **Investigation techniques** include interviews (detailed information from key stakeholders), questionnaires (broad input from distributed groups), observation (first-hand understanding of current processes), and documentation review (analyzing existing data flows and storage)
- **Five core steps:** problem definition, system investigation (data collection), modeling (visual process/data flow representation), analysis (identifying inefficiencies and gaps), and validation/reporting
- **Documentation tools** include data flow diagrams (DFDs), entity-relationship diagrams (ERDs), organization charts, and process flowcharts
- **Structured vs. object-oriented** approaches provide different frameworks for decomposing and modeling system components
- **Deliverables** include system requirements documents, feasibility studies, and recommendations for system improvements

**Practical relevance:** Systems analysis is the critical first phase of any IT project. Analysts who can effectively investigate business problems and translate them into technology requirements are highly valued in consulting, enterprise IT, and software development organizations. These skills are directly applicable to the team projects in ISA 441.

### SDLC

**Definition:** The Software Development Life Cycle (SDLC) is a structured process that defines the phases for planning, creating, testing, deploying, and maintaining software systems. It provides a systematic framework ensuring software meets user needs and quality standards.

- **Core phases** include planning, feasibility analysis/requirements, system design, implementation/coding, testing, deployment, and maintenance
- **Waterfall model** follows a linear, sequential approach where each phase must be completed before the next begins; best suited for projects with well-defined, stable requirements
- **Agile model** uses iterative cycles (sprints) with continuous feedback and adaptation; ideal for complex projects with evolving requirements and the need for frequent stakeholder input
- **Other models** include Spiral (risk-driven iterations), V-Model (verification and validation paired with each phase), RAD (rapid application development), and DevOps (continuous integration/delivery)
- **Choosing a model** depends on project size, requirement stability, team structure, risk tolerance, and stakeholder involvement needs

**Practical relevance:** The SDLC provides the overarching framework for every software and systems development project. Understanding different SDLC models enables teams to select the right approach for their project context, which is a core competency for systems analysts and IT consultants in ISA 441.

### Requirements gathering

**Definition:** Requirements gathering (elicitation) is the process of collecting, documenting, and validating the needs and expectations of stakeholders to define what a system must do. It is the foundation upon which all subsequent design and development decisions are built.

- **Key techniques** (per BABOK): interviews, questionnaires/surveys, observation, brainstorming, focus groups, document analysis, prototyping, requirements workshops, and interface analysis
- **Interviews** allow analysts to dig into stakeholders' knowledge to understand business processes and create system requirements documents (SRDs); can be structured, semi-structured, or informal
- **Requirements workshops** bring stakeholders together in structured, time-boxed sessions to collaboratively elicit, refine, and prioritize requirements when time is limited
- **Use cases** describe how users interact with the system to achieve specific goals, providing scenarios that validate functional requirements
- **Recommended approach:** begin with stakeholder interviews, then run requirements workshops, and validate with use cases and acceptance test criteria

**Practical relevance:** Poor requirements are a leading cause of project failure. Effective requirements gathering ensures the development team builds the right system and reduces costly rework. In ISA 441, students practice these techniques as part of team-based consulting projects for real or simulated clients.

### System design

**Definition:** System design is the process of defining the architecture, components, modules, interfaces, and data flows of a system to satisfy specified requirements. It translates the "what" from requirements analysis into the "how" of implementation using visual modeling tools like UML.

- **Logical design** focuses on the abstract structure of the system (what it does) using class diagrams, object diagrams, and component diagrams independent of technology choices
- **Physical design** maps the logical design onto actual hardware and software infrastructure using deployment diagrams showing servers, nodes, and network connections
- **UML (Unified Modeling Language)** provides standardized diagram types including use case diagrams, class diagrams, sequence diagrams, activity diagrams, and deployment diagrams
- **The 4+1 architectural view model** organizes system design into Logical, Physical, Process, Development, and Use Case views for comprehensive coverage
- **Design patterns** (MVC, Observer, Factory, Singleton) provide reusable solutions to common design problems

**Practical relevance:** System design bridges the gap between requirements and implementation. Producing clear, well-structured design documents ensures developers understand what to build and how components fit together. UML proficiency is a key skill for systems analysts, architects, and IT consultants working on enterprise projects in ISA 441.

### IT consulting

**Definition:** IT consulting is a professional advisory practice that helps organizations optimize their technology ecosystems, solve business problems through technology solutions, and manage digital transformation initiatives. Consultants combine industry expertise with proven methodologies to deliver measurable outcomes.

- **Methodology** follows structured, repeatable frameworks focusing on people, process, technology, and data when evaluating and recommending solutions
- **Key service areas** include IT strategy and roadmapping, cybersecurity, cloud migration, software selection and implementation, data analytics, and digital transformation
- **Engagement lifecycle:** discovery and assessment, requirements definition, solution design, implementation support, and post-implementation review
- **Skills required** include business analysis, stakeholder management, communication, technical knowledge, problem-solving, and project management
- **Deliverables** typically include current-state assessments, gap analyses, technology roadmaps, RFP development, vendor evaluations, and implementation plans

**Practical relevance:** IT consulting is a major career path for ISA graduates. The consulting skills practiced in ISA 441 -- understanding business problems, analyzing systems, proposing technology solutions, and presenting recommendations to stakeholders -- are directly applicable whether working at a consulting firm or serving as an internal technology advisor within an organization.

### Business problem solving

**Definition:** Business problem solving is a structured analytical process for identifying, diagnosing, and resolving organizational challenges through systematic investigation and technology-enabled solutions. It combines critical thinking frameworks with data-driven analysis to move from symptom identification to root cause discovery and actionable recommendations. Effective problem solving requires both analytical rigor and creative thinking to propose solutions that are feasible, cost-effective, and aligned with business objectives.

- **Root Cause Analysis**: Techniques such as the 5 Whys method (repeatedly asking "Why?" to drill past symptoms to underlying causes) and the Fishbone Diagram (Ishikawa Diagram) visually organize potential contributing factors across categories like people, process, technology, and environment to systematically identify the true source of a problem.
- **DMAIC Framework**: Borrowed from Six Sigma, the Define-Measure-Analyze-Improve-Control cycle provides a disciplined, data-driven approach that moves from problem definition through quantitative measurement and analysis to implementing and sustaining improvements.
- **Problem Framing**: Clearly articulating the problem using SMART criteria (Specific, Measurable, Achievable, Relevant, Time-bound) ensures that all stakeholders share a common understanding of the issue scope, success metrics, and constraints before solutions are proposed.
- **Data-Driven Decision Making**: Leveraging data analytics, dashboards, and business intelligence tools provides objective evidence to support diagnoses and validate proposed solutions, reducing reliance on assumptions or anecdotal evidence.
- **Stakeholder Analysis**: Mapping who is affected by the problem, who has decision-making authority, and whose input is needed ensures that proposed solutions account for organizational dynamics, change readiness, and user adoption challenges.
- **Solution Evaluation**: Comparing alternative solutions using criteria such as cost-benefit analysis, risk assessment, technical feasibility, and alignment with strategic goals ensures the recommended approach delivers maximum value. Consulting frameworks like MECE (Mutually Exclusive, Collectively Exhaustive) help structure the evaluation exhaustively.

**Practical Applications:** In IT consulting engagements, business problem solving is used to diagnose why a client's current system is failing to meet operational needs and to develop a structured recommendation for improvement -- for example, applying root cause analysis to recurring data quality issues before proposing a new data governance platform, or using DMAIC to quantify process inefficiencies before recommending an automation solution.

### Project planning

**Definition:** Project planning is the process of defining project scope, objectives, tasks, timelines, resources, and budgets to create a comprehensive roadmap for successful project execution. It is the second of five core project lifecycle phases (Initiation, Planning, Execution, Monitoring and Control, Closure) and produces the foundational documents that guide all subsequent project work. Effective planning ensures projects are completed on time, within budget, and to the required quality standards.

- **Work Breakdown Structure (WBS)**: The WBS hierarchically decomposes the project into deliverables, sub-deliverables, and work packages. It can be structured by phase, by deliverable, or by responsible organizational unit, and serves as the foundation for scheduling, budgeting, and resource assignment.
- **Scheduling Techniques**: PERT (Program Evaluation and Review Technique) estimates task durations and identifies critical path activities to optimize schedules, while Gantt charts provide visual timelines depicting tasks, milestones, and dependencies that enable managers to track progress and manage resources efficiently.
- **Change Control**: Late changes are a major source of disruption leading to schedule slippage, cost overruns, and rework. A formal change control and change management system must be established early to evaluate, approve, and document all scope or requirement modifications.
- **Risk Management**: The risk register identifies potential threats and opportunities, assesses their probability and impact, and defines mitigation strategies and contingency plans to proactively address issues before they derail the project.
- **Estimation and Budgeting**: Techniques include analogous estimation (based on similar past projects), parametric estimation (using statistical models), and bottom-up estimation (aggregating individual work package estimates) to produce realistic cost and duration forecasts.
- **Communication Planning**: Maintaining open and transparent communication with team members and stakeholders through a structured communication plan helps ensure alignment, manage expectations, and facilitate timely decision-making throughout the project lifecycle.

**Practical Applications:** In ISA 441, student teams develop comprehensive project plans for building information systems, applying the same planning discipline used in professional IT consulting engagements -- from creating a project charter and WBS for a client's new system to establishing Gantt-based schedules with milestones, risk registers, and formal change control processes that mirror industry best practices.
