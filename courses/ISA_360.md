### Data warehousing

A data warehouse is a centralized repository that stores integrated data from multiple sources, organized and optimized for analytical querying and reporting rather than transactional processing (OLAP vs. OLTP). Data is structured using schemas -- most commonly the star schema (a central fact table surrounded by denormalized dimension tables for fast queries) or the snowflake schema (dimension tables normalized into sub-tables to reduce redundancy). Fact tables contain quantitative measurements (sales amounts, quantities) while dimension tables hold descriptive attributes (dates, products, customers).

- **Star schema**: Central fact table directly connected to dimension tables; denormalized design enables fast query performance; simple and intuitive for business users; trades storage efficiency for speed
- **Snowflake schema**: Dimension tables normalized into multiple related sub-tables; reduces data redundancy; easier to maintain when dimensions change frequently; slightly more complex queries due to additional joins
- **Key concepts**: Fact tables (measures and foreign keys), dimension tables (descriptive attributes), slowly changing dimensions (SCD Type 1, 2, 3 for tracking historical changes), data marts (subject-specific subsets)
- **Architecture layers**: Source systems, staging area, ETL processing, data warehouse storage, OLAP cubes or semantic layers, and reporting/BI tools

**Practical relevance**: Data warehouses are the backbone of enterprise business intelligence, enabling organizations to consolidate data from disparate systems and perform historical analysis, trend identification, and strategic reporting at scale.

### Distributed databases

A distributed database is a system where data is stored across multiple physical or virtual servers (nodes) connected by a network, rather than on a single machine. This architecture enables horizontal scalability, fault tolerance, and geographic distribution of data. The two core mechanisms are partitioning (splitting data across nodes so each node handles a subset) and replication (storing multiple copies of data for availability and redundancy). Distributed databases must manage the CAP theorem tradeoff: a system can provide at most two of Consistency, Availability, and Partition tolerance simultaneously.

- **Partitioning strategies**: Horizontal partitioning/sharding (rows distributed by key range or hash); vertical partitioning (columns split across nodes); each partition can be placed on different nodes to distribute query load
- **Replication models**: Active-passive (writes go to primary, reads served by replicas); active-active (multiple nodes accept writes); synchronous replication (strong consistency) vs. asynchronous (eventual consistency with lower latency)
- **Architecture types**: Client/server, peer-to-peer, and federated/multi-database systems; nodes communicate to coordinate queries and transactions across the distributed system
- **Challenges**: Network latency, distributed transactions (two-phase commit), data consistency, conflict resolution in multi-master setups, and partition management overhead

**Practical relevance**: Distributed databases power large-scale web applications, cloud platforms, and big data systems where single-server databases cannot handle the volume, velocity, or availability requirements. Technologies like Cassandra, CockroachDB, and Google Spanner exemplify modern distributed database systems.

### Advanced SQL

Advanced SQL extends beyond basic SELECT/JOIN/WHERE queries to include powerful analytical constructs such as window functions, Common Table Expressions (CTEs), subqueries, and procedural extensions. Window functions perform calculations across a set of rows related to the current row without collapsing them into a single output (unlike GROUP BY), enabling running totals, rankings, and moving averages. CTEs provide named temporary result sets that make complex queries readable and maintainable, and support recursive queries for hierarchical data.

- **Window functions**: ROW_NUMBER(), RANK(), DENSE_RANK() for ranking; SUM/AVG/COUNT OVER (PARTITION BY ... ORDER BY ...) for running aggregations; LAG/LEAD for accessing previous/next rows; NTILE for distribution into buckets
- **Common Table Expressions (CTEs)**: WITH clause creates named temporary result sets; improves readability of complex multi-step queries; recursive CTEs traverse hierarchical data (org charts, bill of materials, threaded comments)
- **Subqueries**: Correlated subqueries reference outer query values; scalar subqueries return single values; EXISTS/IN for semi-joins; used for dynamic filtering and computed columns
- **Additional techniques**: CASE expressions for conditional logic, COALESCE/NULLIF for null handling, PIVOT/UNPIVOT for reshaping data, stored procedures for reusable logic, and query optimization using EXPLAIN plans and indexes

**Practical relevance**: Advanced SQL skills are essential for data engineers and analysts working with data warehouses, enabling efficient extraction of complex business metrics, time-series analysis, and hierarchical data traversal directly within the database engine.

### Hadoop

Apache Hadoop is an open-source framework for distributed storage and processing of large datasets across clusters of commodity hardware. Its core modules include HDFS (Hadoop Distributed File System) for storage, MapReduce for parallel computation, YARN for resource management and job scheduling, and Hadoop Common for shared utilities. HDFS splits files into large blocks (typically 128-256 MB) distributed across nodes with default triple replication for fault tolerance. MapReduce processes data in two phases: Map (splits data into key-value pairs processed in parallel) and Reduce (aggregates intermediate results into final output).

- **HDFS**: Write-once, read-many file system; NameNode manages metadata, DataNodes store blocks; automatic replication ensures data survives node failures; optimized for large sequential reads
- **MapReduce**: Programmer defines Map function (filters and sorts) and Reduce function (aggregates); framework handles distribution, parallelization, fault tolerance, and scheduling automatically
- **YARN**: Yet Another Resource Negotiator decouples resource management from processing; enables multiple processing engines (Spark, Tez, Flink) to share the same Hadoop cluster
- **Ecosystem tools**: Hive (SQL-like queries), Pig (scripting), HBase (NoSQL column store), Sqoop (data import/export), Flume (log collection), Oozie (workflow scheduling), Zookeeper (coordination)

**Practical relevance**: Hadoop revolutionized big data processing by making it economically feasible to store and analyze petabytes of data on commodity hardware. While newer tools like Spark have superseded MapReduce for many workloads, HDFS and the Hadoop ecosystem remain foundational in enterprise data architectures.

### NoSQL

NoSQL (Not Only SQL) databases are non-relational database systems designed to handle large volumes of unstructured or semi-structured data with flexible schemas and horizontal scalability. Unlike relational databases that enforce rigid table structures, NoSQL databases allow varied data models optimized for specific access patterns. There are four main types, each suited to different use cases: document stores, key-value stores, wide-column stores, and graph databases.

- **Document databases** (MongoDB, CouchDB): Store data as JSON/BSON documents with nested fields and arrays; flexible schema allows different structures per document; query by any field within the document; ideal for content management, catalogs, and user profiles
- **Key-value stores** (Redis, DynamoDB): Simplest model -- each record is a key mapped to a value; extremely fast lookups by key; ideal for caching, session management, and real-time leaderboards
- **Wide-column stores** (Cassandra, HBase): Organize data into column families with unique row keys; optimized for writing and reading large datasets across distributed clusters; ideal for time-series data, IoT, and logging
- **Graph databases** (Neo4j, Amazon Neptune): Model data as nodes and edges representing entities and relationships; efficient traversal of connected data; ideal for social networks, recommendation engines, and fraud detection

**Practical relevance**: NoSQL databases power modern web-scale applications where relational databases face limitations in scalability, flexibility, or performance. Choosing the right NoSQL type depends on data structure, query patterns, consistency requirements, and scaling needs.

### Big Data

Big Data refers to datasets so large, fast-moving, or complex that traditional data processing tools cannot handle them effectively. It is defined by the 5 Vs: Volume (terabytes to exabytes of data), Velocity (real-time or near-real-time data generation), Variety (structured, semi-structured, and unstructured formats), Veracity (data quality and trustworthiness), and Value (actionable business insights derived from analysis). Big Data encompasses the technologies, practices, and architectures needed to capture, store, process, and analyze these massive datasets.

- **Volume**: Data measured in terabytes, petabytes, or exabytes; generated by sensors, social media, transactions, logs, and digital interactions at unprecedented scale
- **Velocity**: Data produced and consumed at high speed; real-time streaming from IoT devices, financial markets, social feeds, and clickstreams demands immediate or near-immediate processing
- **Variety**: Includes structured data (databases, spreadsheets), semi-structured (JSON, XML, logs), and unstructured data (text, images, video, audio) -- much of which does not fit traditional row-column formats
- **Key technologies**: Hadoop (distributed storage and batch processing), Spark (fast in-memory processing), Kafka (stream processing), cloud platforms (AWS, Azure, GCP) providing managed big data services

**Practical relevance**: Big Data analytics enables organizations to uncover patterns, trends, and associations in massive datasets that would be invisible in smaller samples. Applications span personalized marketing, predictive maintenance, genomics research, smart cities, and financial risk management.

### ETL processes

ETL (Extract, Transform, Load) is a data integration process that combines data from multiple sources, cleans and restructures it, and loads it into a target system such as a data warehouse or data lake. The Extract phase copies raw data from source systems (databases, APIs, files) into a staging area. The Transform phase applies cleaning, validation, deduplication, format standardization, aggregation, and business logic. The Load phase inserts the processed data into the final target system for analysis and reporting.

- **Extract**: Pull data from heterogeneous sources including relational databases, flat files, APIs, cloud services, and legacy systems; strategies include full extraction (all data) and incremental extraction (only changed records)
- **Transform**: Data cleaning (handle nulls, remove duplicates, fix errors), type conversions, business rule application, joins and lookups, aggregations, surrogate key generation, and slowly changing dimension processing
- **Load**: Insert data into target tables; strategies include full refresh (replace all data) and incremental load (append or merge changes); bulk loading for performance, with error handling and logging
- **ETL vs. ELT**: Traditional ETL transforms data before loading; modern ELT (Extract, Load, Transform) loads raw data first and transforms it within the target system using its processing power, common in cloud data warehouses like Snowflake and BigQuery

**Practical relevance**: ETL processes are the backbone of data warehousing and business intelligence, ensuring that decision-makers have access to clean, consistent, and timely data. Data engineers design and maintain ETL pipelines as a core job function.
