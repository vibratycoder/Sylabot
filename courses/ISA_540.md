### Relational databases

Relational databases organize data into structured tables (relations) with rows and columns, connected through defined relationships using primary and foreign keys. Based on Edgar Codd's relational model, they enforce data integrity through constraints and provide a mathematically grounded approach to data storage and retrieval.

- **Key concepts:** Tables (relations), rows (tuples/records), columns (attributes/fields), primary keys (unique row identifiers), foreign keys (cross-table references), indexes (performance optimization)
- **ACID properties:** Atomicity (all-or-nothing transactions), Consistency (data remains valid), Isolation (concurrent transactions don't interfere), Durability (committed data persists)
- **Normalization:** Process of organizing tables to reduce redundancy; normal forms (1NF, 2NF, 3NF, BCNF) progressively eliminate data anomalies
- **Popular systems:** PostgreSQL (open-source, feature-rich), MySQL (widely deployed), Microsoft SQL Server (enterprise), Oracle Database (enterprise), SQLite (embedded)
- **Practical relevance:** Relational databases remain the backbone of enterprise data storage for transactional systems, financial records, CRM, and ERP platforms. Their strong consistency guarantees and mature tooling make them the default choice when data integrity is paramount.

### SQL

Structured Query Language (SQL) is the standard language for managing and querying relational databases. It provides a declarative syntax for creating, reading, updating, and deleting data, as well as defining database structures and controlling access permissions.

- **Core operations (CRUD):** SELECT (read), INSERT (create), UPDATE (modify), DELETE (remove); WHERE clause filters rows; JOIN combines tables
- **Advanced querying:** Subqueries, Common Table Expressions (CTEs), window functions (ROW_NUMBER, RANK, LAG, LEAD), aggregate functions (SUM, COUNT, AVG, GROUP BY, HAVING)
- **Data definition:** CREATE TABLE, ALTER TABLE, DROP TABLE; defining constraints (PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE, CHECK)
- **Performance optimization:** Indexing strategies, query execution plans (EXPLAIN), avoiding N+1 queries, partitioning large tables
- **Practical relevance:** SQL is the most essential technical skill for data professionals, consistently ranking as the top requirement in analytics job postings. It enables direct interaction with organizational data stores for reporting, analysis, and pipeline construction, and remains relevant even as new technologies emerge.

### NoSQL

NoSQL (Not Only SQL) databases are non-relational data stores designed to handle unstructured, semi-structured, or rapidly changing data at scale. They sacrifice some relational guarantees in favor of flexibility, horizontal scalability, and performance for specific workload patterns.

- **Database types:** Document stores (MongoDB -- stores JSON-like documents), key-value stores (Redis -- fast caching and lookups), wide-column stores (Cassandra -- high write throughput), graph databases (Neo4j -- relationship-focused queries)
- **CAP theorem:** Distributed systems can guarantee at most two of three properties: Consistency, Availability, and Partition tolerance; NoSQL databases make different tradeoffs
- **Schema flexibility:** NoSQL databases typically allow schema-less or schema-on-read designs, enabling agile development and handling of varied data structures
- **Scalability:** Designed for horizontal scaling (adding more nodes) rather than vertical scaling; Cassandra and MongoDB can distribute data across hundreds of nodes
- **Practical relevance:** NoSQL databases power modern applications including social media platforms, IoT sensor data collection, real-time analytics, content management systems, and e-commerce product catalogs. Choosing between SQL and NoSQL depends on data structure, consistency requirements, and scale needs.

### Vector databases

Vector databases are specialized data stores designed to efficiently index, store, and query high-dimensional vector embeddings -- numerical representations of data generated by AI models. They enable similarity-based search by finding vectors that are closest to a query vector in high-dimensional space.

- **Key concepts:** Vector embeddings (dense numerical representations of text, images, or other data), similarity metrics (cosine similarity, Euclidean distance, dot product), approximate nearest neighbor (ANN) search algorithms
- **Popular systems:** Pinecone (managed cloud), Weaviate (open-source), Milvus (open-source), Chroma (lightweight), pgvector (PostgreSQL extension), Qdrant
- **Indexing algorithms:** HNSW (Hierarchical Navigable Small World), IVF (Inverted File Index), Product Quantization; these enable fast approximate search across millions of vectors
- **RAG integration:** Vector databases are central to Retrieval-Augmented Generation, storing document embeddings that LLMs query to ground responses in factual, up-to-date information
- **Practical relevance:** Vector databases are essential infrastructure for modern AI applications including semantic search, recommendation engines, image similarity, chatbots with knowledge bases, and drug discovery. Their importance has surged with the rise of LLMs and generative AI.

### Data management for AI

Data management for AI encompasses the strategies, processes, and technologies required to collect, store, govern, and deliver high-quality data that AI systems need for training, inference, and continuous improvement. It bridges traditional data management practices with the unique demands of machine learning workflows.

- **Data quality:** Ensuring accuracy, completeness, consistency, and timeliness; data profiling and validation; handling missing values, duplicates, and outliers
- **Data governance:** Metadata management, data lineage tracking, access controls, privacy compliance (GDPR, CCPA), responsible AI practices, bias detection and mitigation
- **Feature stores:** Centralized repositories for computing, storing, and serving ML features; ensure consistency between training and serving environments (Feast, Tecton)
- **Data versioning:** Tracking changes to datasets over time; tools like DVC (Data Version Control) and LakeFS enable reproducible experiments
- **Practical relevance:** AI systems are only as good as their data -- the "garbage in, garbage out" principle is amplified in machine learning. Organizations that invest in robust data management infrastructure see faster model development, more reliable predictions, and better regulatory compliance.

### Database design

Database design is the process of defining the structure, relationships, and constraints of a database to efficiently store and retrieve data while maintaining integrity and performance. Good design prevents data anomalies, reduces redundancy, and ensures the database can evolve with changing requirements.

- **Design process:** Requirements analysis, conceptual design (entity-relationship modeling), logical design (schema definition, normalization), physical design (indexing, partitioning, storage optimization)
- **ER modeling:** Entity-Relationship diagrams represent entities (tables), attributes (columns), and relationships (one-to-one, one-to-many, many-to-many) visually before implementation
- **Normalization vs. denormalization:** Normalization reduces redundancy and anomalies; denormalization intentionally introduces redundancy to improve read performance for analytics workloads
- **Modern considerations:** Designing for both OLTP (transactional) and OLAP (analytical) workloads; schema design for NoSQL databases; data lake and data warehouse architectures (star schema, snowflake schema)
- **Practical relevance:** Poor database design leads to data inconsistencies, slow queries, and systems that are difficult to maintain or scale. For AI practitioners, understanding database design ensures efficient data pipelines and proper data modeling that supports both operational systems and analytical workloads.
